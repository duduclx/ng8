J'ai appliqué le même style (genre container) à mes pages ... et dans chaque css, j'ai le même code.

Un composant étant indépendant du reste, il l'est notemment avec le style, bonne et mauvaise chose. Mais revenons à notre histoire de data-binding.

Derrière de mot bizarre, c'est simplement le fait d'utiliser une variable dans un template, mais aussi, depuis le template, pouvoir agir sur le composant.

dans chapter-four.component.ts, je vais créer, à l'intérieur du export class, des propriétés prénom et nom, de type privée (private) et typées string.

ce qui donne

"private prenom: string;"
"private nom: string;"

dans la fonction ngOnInit(), j'affecte des valeurs à ces propriétés.

Pour les utiliser dans le template, j'utilise l'interpolation:

{{ '{{' }} mapropriété }}

Ce qui donne mon prenom est: {{ prenom }} et mon nom: {{ nom }}

ben faut aller voir le code pour comprendre, mais j'ai écrit:

Ce qui donne mon prenom est: {{ '{{' }} prenom }} et mon nom: {{ '{{' }} nom }}

Maintenant, depuis le template, créons un bouton pour créer un évènement, et changer les valeurs des propriétés:

change julien en toto {{ prenom }}

j'ai appelé un event de type click avec (click) et je lui est passé en argument une fonction onClick() écrite dans le composant, qui change le prenom via this.prenom = 'toto';

je vais ajouter aussi, que si this.prenom == 'toto' alors le click change this.prenom en julien ...

C'était sympa, bien que très bref ... car on peut faire plein de chose, mais à ce niveau, on va en rester là.

On va rester dans le data-binding, mais en ajoutant une notion parent-child (parent - enfant).
